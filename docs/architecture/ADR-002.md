Readiness as a Derived Decision Layer

Status: Accepted
Date: 2026-03-13

Context

Планируемые operational modules включают:

StockSnapshot

Order

OrderLine

PickTask

PickItem

Но наивный подход — генерировать picking tasks напрямую из order lines — делает систему неявной и плохо объяснимой. В такой модели готовность заказа определяется побочно: “если task создался — значит ready”.

Это плохая архитектура.

Уже сейчас system overview показывает, что readiness и picking — отдельный будущий bounded context, а не просто экранный статус.

Decision

Readiness выделяется в отдельный вычислительный контур.

Readiness engine использует:

storage truth

product location roles

layout/spatial constraints

stock availability

placement validity

operational rules

Readiness engine производит:

ready

partially_ready

blocked

shortage

invalid_mapping

needs_replenishment

Pick execution не вычисляет readiness самостоятельно

Правило:

PickTask создаётся только из readiness result, а не напрямую из OrderLine.

Why

Это позволяет системе не просто принять решение, но и объяснить его.

Пример reason model:

inventory absent

inventory exists only in reserve

cell blocked

invalid pick mapping

split inventory across containers

replenishment required

Consequences
Positive

появляется explainability

readiness можно пересчитывать независимо

replenishment можно строить поверх reason codes

picking становится execution layer, а не местом скрытой бизнес-логики

Negative

добавляется отдельный модуль и lifecycle

нужна reason taxonomy

часть пользователей может считать это “лишней сложностью”

Rejected alternatives
1. Генерировать pick tasks напрямую из orders

Rejected, потому что это смешивает planning и execution.

2. Хранить readiness как ручной статус order line

Rejected, потому что readiness должен быть derived result, а не ручной флаг.
