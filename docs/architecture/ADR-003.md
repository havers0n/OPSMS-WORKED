Domain Events over Audit-Only Logging

Status: Accepted
Date: 2026-03-13

Context

Текущая система уже использует operation_events в layout lifecycle flow для audit trail при save/validate/publish. Это полезно, но этого недостаточно для operational WMS.

Audit logging отвечает на вопрос:

кто и когда что-то изменил?

Но не отвечает хорошо на вопросы:

что именно произошло в терминах домена?

какая цепочка событий привела к проблеме?

можно ли replay’ить поведение?

можно ли строить explainable analytics?

Decision

Система принимает event-native подход для ключевых доменных переходов.

Вводится domain event model

Минимальные события:

LayoutPublished

ContainerPlaced

ContainerMoved

InventoryAdjusted

ImportPublished

ReadinessEvaluated

PickTaskCreated

PickItemConfirmed

Event model хранит:

aggregate_type

aggregate_id

event_type

payload

occurred_at

caused_by

correlation_id

Audit logging остаётся, но не считается достаточным

Правило:

Audit log — это вспомогательный след, а domain events — основной semantic history layer.

Consequences
Positive

replay/debug становится проще

analytics можно строить на событиях, а не на догадках по текущим таблицам

AI insights можно строить на реальном процессе

легче исследовать bottlenecks и route compliance

Negative

больше write complexity

нужно поддерживать event schema versioning

возможен риск шумного event stream без дисциплины

Rejected alternatives
1. Оставить только row-level audit

Rejected, потому что row mutation не равна domain event.

2. Сразу строить сложную event bus architecture

Rejected на текущем этапе. Сначала нужна простая нормализованная event table.
