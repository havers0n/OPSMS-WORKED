Status: Accepted
Date: 2026-03-13

Context

Текущая layout-core архитектура уже использует несколько слоёв состояния с разными владельцами и разной степенью авторитетности:

Supabase Auth session

BFF-resolved workspace context

TanStack Query cached server state

persisted layout draft in Postgres

local editor working state in Zustand

derived UI state and validation previews

Эти слои уже описаны в state map, и там прямо зафиксировано, что похожая shape данных не означает одинаковый уровень правды.

Проблема в том, что без формальной иерархии state truth разработчики начинают:

считать Query cache источником правды,

считать local editor draft persisted truth,

использовать cached validation как truth для dirty local state,

принимать lifecycle-решения на основе UI-derived flags вместо persisted state.

Это создаёт скрытую архитектурную путаницу и приводит к ошибкам на границах layout lifecycle.

Decision

Система принимает явную иерархию правды состояния.

1. Auth truth

Canonical owner: Supabase Auth session

Отвечает за:

authenticated user identity

bearer token

current auth session

Правило:

frontend local auth state является только in-memory projection auth session

route/UI state не может считаться auth truth.

2. Workspace truth

Canonical owner: BFF /api/me

Отвечает за:

current tenant context

memberships

workspace-scoped identity projection

Правило:

tenant/workspace context не определяется UI store или route params

frontend хранит только local copy resolved workspace session

auth reset обязан очищать workspace-scoped local state.

3. Persisted editing truth

Canonical owner: Postgres persisted draft (layout_versions + child tables)

Отвечает за:

saved editable layout state

active draft for floor

persisted rack/face/section/level tree

Правило:

persisted draft является authoritative editing truth после успешного save

published layout не редактируется in place

layout lifecycle ownership живёт в DB/RPC, а не в UI.

4. Local working truth

Canonical owner: editor Zustand store

Отвечает за:

unsaved in-progress edits

current browser-session working copy

selection, hover, zoom, placement mode, draft dirty state

Правило:

local editor draft является truth только для unsaved local work

local dirty state не переопределяет persisted truth

re-fetch persisted draft не должен неявно маскировать divergence между local and saved state.

5. Cached server projection

Canonical owner: TanStack Query cache

Отвечает за:

transport-level cache of server reads

sites, floors, active layout draft, published summary, cached validation result

Правило:

Query cache не является business truth

cache is disposable

invalidation refreshes projection, not source of truth.

6. Derived UI state

Canonical owner: none; derived only

Включает:

action enablement

inspector summaries

validation preview

address preview

status labels

setup state decisions

Правило:

derived UI state никогда не считается canonical truth

derived state не должен становиться входом для persistence decisions без явной canonical source.

Truth hierarchy

Когда слои расходятся, применяется следующий порядок:

Supabase Auth session — truth for auth identity

BFF /api/me — truth for workspace context

Postgres persisted draft/published records — truth for saved layout state

Zustand editor store — truth only for current unsaved local edits

Query cache — disposable cached server projection

Derived UI state — never authoritative truth.

Operational rules
Rule 1

Lifecycle decisions must use persisted state, not UI projections.

Примеры:

publish eligibility опирается на persisted draft + validation result, а не на button state

active draft existence определяется persistence/RPC, а не presence of local draft object

Rule 2

Cached validation is not truth for dirty local draft.

Если local editor state dirty, cached validation может использоваться только как stale reference, но не как proof that current draft is valid. Это уже зафиксировано в runtime flow and state split.

Rule 3

UI selection state does not create domain truth.

activeSiteId и activeFloorId выбирают контекст экрана, но не определяют, какие сущности существуют в системе.

Rule 4

Published truth and editable truth remain distinct.

Published layout — authoritative spatial truth for operations.
Persisted draft — editable truth for next change cycle.
Они не смешиваются в одну state model.

Consequences
Positive

упрощается reasoning о source of truth

снижается риск утечки UI assumptions в lifecycle logic

local edit model остаётся совместимой с persisted lifecycle

легче строить дальнейшие operational modules поверх stable truth boundaries

Negative

появляется дополнительная дисциплина на границах состояния

разработчикам нужно явно думать, с каким слоем они работают

часть seemingly convenient shortcuts becomes disallowed

Rejected alternatives
1. Treat Query cache as application truth

Rejected, потому что cache — это transport projection, не canonical state.

2. Treat local editor store as the main layout truth

Rejected, потому что local store содержит unsaved working copy и может расходиться с persisted draft.

3. Collapse published and draft layout into one mutable state

Rejected, потому что это ломает layout lifecycle model draft -> published -> archived и размывает operational spatial truth.
