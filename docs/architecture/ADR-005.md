Status: Accepted
Date: 2026-03-13

Context

Текущая архитектура уже задумывает BFF как thin orchestration layer over Supabase tables and RPC. Это прямо зафиксировано в system overview и API contract map:

BFF authenticates requests

resolves workspace context

exposes REST endpoints

calls Supabase tables and RPC

normalizes errors

При этом business truth должен жить в Postgres/RPC/domain contracts, а не в BFF.

Одновременно invariant matrix честно фиксирует, что эта граница пока соблюдается частично: в текущем layout flow она уважается, но не жёстко enforced repo-wide.

Проблема типовая: если не зафиксировать boundary formally, BFF быстро превращается в скрытый backend-with-business-logic:

lifecycle rules начинают жить в route handlers

transport validation смешивается с domain validation

одна и та же бизнес-логика дублируется в BFF и DB

truth fragments расходятся между HTTP layer и persistence layer

Это особенно опасно для WMS, потому что lifecycle, readiness, import discipline и storage invariants должны быть предсказуемыми и transactional.

Decision

BFF принимается как thin boundary layer, а не как primary domain logic owner.

BFF responsibilities

BFF должен:

authenticate request using bearer token

resolve workspace/session context

validate transport shape of path/query/body

call Supabase tables and RPC

map persistence responses into transport DTOs

normalize and classify errors

log request lifecycle and request-scoped metadata.

BFF non-responsibilities

BFF не должен быть местом, где живут:

authoritative lifecycle rules

transactional publish/save/readiness/import semantics

data integrity invariants that belong to DB/domain

duplicated canonical enums/rules that can drift from domain truth

operational truth derived independently from persistence layer

Boundary rule
Canonical business truth lives in:

Postgres schema

constraints

RPC/functions

shared domain contracts in packages/domain

BFF owns only:

transport

orchestration

mapping

auth/workspace resolution.

Examples
Allowed in BFF

reject missing bearer token with 401

map PGRST116 to 404 NOT_FOUND

validate that siteId path param is UUID

call publish_layout_version(layout_version_id, actor_id)

map DB rows into LayoutDraft DTO

return normalized error envelope.

Not allowed in BFF

decide independently whether a layout may publish

recalculate canonical storage truth outside DB/domain

create alternate readiness logic in route handler

encode lifecycle transitions only in TypeScript routes

let BFF validation become looser or semantically different than domain truth on core entities

Последний риск уже виден в contract parity audit: save DTO currently accepts weaker values than downstream invariants for fields like kind, axis, side, slotNumberingDirection, and rotationDeg. Это weakened boundary, а не acceptable long-term state. provided user audit

Validation rule

BFF request validation validates transport shape, not the full semantic truth.

Это уже зафиксировано в API contract map: several semantic constraints are enforced in Postgres, not in Zod.

Из этого следуют два правила:

Rule 1

BFF may validate shape and obvious literal sets early.

Пример:

UUID format

enum membership

required object structure

Rule 2

BFF must not become the only owner of semantic constraints.

Если constraint важен для data truth, он должен быть enforceable in DB/domain even if BFF is bypassed.

Transaction rule

All multi-row or lifecycle-sensitive operations must be owned by RPC/DB transaction boundaries, not by ad hoc BFF orchestration.

Примеры:

create_layout_draft

save_layout_draft

validate_layout_version

publish_layout_version already belong to DB/RPC and must remain there.

Будущие import, readiness, and picking modules должны следовать тому же правилу.

DTO rule

BFF may define transport DTOs, but transport DTOs must not silently redefine domain truth.

Следствия:

transport names can differ for API ergonomics only when mapping is explicit

DTO omission of important lifecycle fields must be intentional and documented

field weakening or loss across BFF boundary is treated as contract drift, not as harmless flexibility

Это уже подтверждается current parity audit findings around dropped, weakened, and partial fields. provided user audit

Consequences
Positive

DB/RPC remains the authoritative lifecycle owner

легче поддерживать transactional integrity

меньше duplication of business rules

легче проверять invariants through SQL/domain tests

BFF remains replaceable as application gateway

Negative

часть логики придётся реализовывать глубже, а не “быстро в route handler”

route code becomes less “smart”

some frontend-visible checks may require explicit domain/DB support instead of local convenience

Review checklist for new endpoints

Для каждого нового BFF endpoint нужно пройти 5 вопросов:

Это transport orchestration или business rule?

Если это business rule, почему она не живёт в domain/RPC?

Можно ли bypass BFF и сохранить invariant?

Есть ли duplication between BFF validation and DB/domain validation?

Не создаёт ли endpoint новый source of truth inside BFF?

Если хотя бы на один вопрос ответ плохой, endpoint boundary считается suspect.

Rejected alternatives
1. Put lifecycle and operational rules in BFF route handlers

Rejected, потому что это разносит truth across HTTP layer and DB and destroys transactional clarity.

2. Make BFF the primary application backend, DB a passive store

Rejected, потому что текущая архитектура уже опирается на Postgres/RPC as authoritative logic owner, и это лучше соответствует layout lifecycle and future operational invariants.

3. Keep BFF validation intentionally loose and rely only on DB failures

Rejected as target state. Early transport rejection may be thin-layer responsibility, but canonical invariant ownership still stays below BFF. Weak BFF validation on core enums is treated as fixable drift, not desired architecture. provided user audit
