Title

Warehouse Domain Roadmap: Floor-Centric Workspace, Storage Core, and Movement History

Context

The current warehouse flow is still too tightly bound to the layout draft lifecycle.

Today the system already has two different kinds of truth:

1. Design truth

- layout versions
- racks
- faces
- sections
- levels
- cells

2. Execution truth

- locations
- container types
- containers
- inventory units
- movements
- derived execution snapshots

The immediate product problem is already visible:

- Placement depends on published cells.
- The /warehouse workspace still gates access on active draft existence.

This creates an incorrect operator workflow and an incorrect system mental model.

At the same time, the execution model is not mature enough yet to introduce a full operational control layer such as:

- movement_events
- cell_blocks
- operational_exceptions

If those concepts are added before the storage core is stable, the model becomes more complex without a reliable execution foundation.

Decision

The warehouse roadmap is adopted in three phases.

Phase 1

Workspace Fix

The workspace becomes floor-centric, not draft-centric.

The primary workspace entity is the floor. Layout editing and operational placement are separate concerns that are both loaded inside the floor workspace.

Target model:

Floor
- draft layout for editing
- published layout for operational structure

Rules:

- canOpenFloor = activeDraft OR latestPublished
- Layout mode is editable only when draft exists
- Layout mode is read-only when only published exists
- Placement mode depends only on published existence
- Placement must not depend on draft existence

Temporary compatibility:

- After publish, the system may auto-create the next draft from the published version.
- This is a transitional UX compatibility mechanism, not the target domain model.

Phase 2

Storage Core Stabilization

Before introducing execution history or exception layers, the execution core must become stable and unambiguous.

Execution core:

- locations
- container_types
- containers
- inventory_units
- movements

Read models:

- inventory_by_location_v
- container_storage_snapshot_v

This phase is about making current warehouse state trustworthy.

Required outcomes:

- executable location semantics are deterministic
- container move semantics are unambiguous
- inventory aggregation is correct
- geometry-to-location linkage is explicit
- snapshot views are trusted read models rather than transitional helpers
- execution RPCs enforce caller identity at the function boundary; caller-supplied
  actor parameters cannot influence the audit trail

The source-of-truth rule for this phase is:

- locations = executable storage truth
- containers = current handling-unit truth
- inventory_units = current inventory truth
- movements = execution transition log
- snapshot views = derived execution reads

Phase 3

Movement History Layer

Only after the storage core is stable, movement history is introduced.

New entity:

- movement_events

Purpose:

- current execution state stays simple
- history becomes explicit
- analytics and audit become possible without redefining current-state truth

Source-of-truth rule:

- locations + containers + inventory_units = current state
- movement_events = historical semantic trace

This enables:

- route analysis
- movement heatmaps
- operator flow analysis
- pick and replenishment optimization
- auditable execution history

Deferred

The following concepts are explicitly deferred beyond this roadmap stage:

- cell_blocks
- operational_exceptions

They belong to a later execution-control layer, after:

- the floor-centric workspace is in place
- the storage core is stable
- movement history exists

Rationale

This ordering is intentional.

Rejected sequence:

- fix workspace
- immediately add movement_events, cell_blocks, operational_exceptions

Why rejected:

- storage semantics are not stable enough yet
- current-state truth would compete with partially introduced history/control concepts
- the model would become larger before becoming clearer

Accepted sequence:

- Phase 1: fix the workspace boundary
- Phase 2: stabilize execution core
- Phase 3: add movement history

This preserves a clean progression:

- first fix access and state boundaries
- then stabilize execution truth
- only then add history

Consequences

Positive

- correct separation between design truth and execution truth
- placement becomes available without artificial draft creation
- clearer operator mental model
- stable foundation before introducing history and analytics
- lower risk of competing sources of truth

Negative

- the editor must support read-only published layout mode
- workspace resolution becomes more explicit
- some temporary compatibility logic may remain until the new workspace model is complete
- more advanced operational control concepts are intentionally delayed

Operational Model Summary

The warehouse domain is structured as:

Floor
- Design world
  - draft layout
  - published layout
- Execution core
  - containers (canonical current_location_id is authoritative)
  - container_placements (rack/canvas geometry projection; not canonical truth)
  - inventory_unit (canonical stock unit; inventory_items is legacy compat)
  - stock_movements (canonical execution history for new flows)
  - storage snapshots
- Execution history (legacy compatibility surface)
  - movement_events

Core principle:

- design = intended structure
- execution core = current operational truth
- execution history = how operational truth changed over time

Exit Criteria

Phase 1 exit criteria

- /warehouse opens when a floor has published layout even without draft
- Placement works without active draft
- Layout is read-only when only published exists

Phase 2 exit criteria

- placement lifecycle is stable
- inventory aggregation is stable
- storage snapshots are trusted
- execution RPCs are hardened: SECURITY DEFINER functions enforce inline
  authorization; direct helper calls are denied to the authenticated role;
  actor identity derives from auth.uid(), not caller-supplied parameters
  (completed in migration 0044, 2026-03-17)

Phase 3 exit criteria

- movement history can be recorded without redefining current placement truth
- current state and history are clearly separated

Rejected Alternatives

1. Keep workspace draft-centric

Rejected because placement is operationally bound to published cells, not to draft existence.

2. Introduce movement and exception layers before stabilizing storage core

Rejected because it expands the domain before current execution truth is mature.

3. Collapse design and execution into one warehouse truth

Rejected because intended structure and actual execution evolve under different rules and lifecycles.
