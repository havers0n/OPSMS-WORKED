Date: 2026-03-15

Title

Core WMS Data Model v1: Executable Locations and Container-First Inventory

Context

The repo already distinguishes spatial truth from storage truth.

Spatial truth today is strong:

- published layout
- racks
- faces
- levels
- generated cells

Execution truth exists, but the conceptual model is still too close to raw published cells.

That is not stable enough for a WMS core that must support:

- receive
- putaway
- transfer
- partial pick
- ship

If execution keeps depending directly on geometry objects, the model will drift into a brittle hybrid where one entity tries to mean:

- spatial slot
- operational location
- current stock carrier
- pick target
- business mapping

Decision

The canonical v1 execution model is:

- `Location`
- `ContainerType`
- `Container`
- `Product`
- `InventoryUnit`
- `Movement`

Core rule:

- geometry slot is spatial truth
- location is executable storage truth
- inventory belongs to containers
- movement records execution transitions

Canonical relationship:

`GeometrySlot -> Location -> Container -> InventoryUnit`

Key invariants:

- inventory cannot exist without a container
- active containers must live in a location except short-lived `in_transit`
- slot and location are not the same entity
- single-container locations cannot hold multiple active containers
- disabled locations reject new placements
- fit must validate both dimensions and weight

Consequences

Positive

- geometry stays resilient under layout evolution
- execution model becomes container-first instead of slot-first
- partial pick becomes first-class instead of awkward
- non-rack operational locations become natural, not exceptional
- API and UX can evolve around executable locations rather than raw geometry rows

Negative

- the current implementation is only partially aligned
- schema and API will need an explicit convergence phase
- some existing docs that describe storage truth via cell-centric placement need refinement

Relationship to existing docs

- `ADR-001` remains valid in principle: spatial truth and storage truth stay separate
- `ADR-006` remains valid in roadmap direction, but Phase 2 storage-core stabilization is now explicitly location-centered
- the detailed canonical reference is `core-wms-data-model-v1.md`

Practical rule

Any new schema, API, or UX decision touching storage semantics should be evaluated against this ADR first.
