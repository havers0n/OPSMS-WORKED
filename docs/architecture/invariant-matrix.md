# Invariant Matrix

## Purpose

This document is intentionally ruthless.

It exists to separate:

- invariants that are actually enforced
- invariants that are only partially guarded
- invariants that are currently declarative architecture, not runtime truth

If an invariant is not backed by a real enforcement point and real tests, it should not be described as protected.

Snapshot date: `2026-03-18` (updated for Stage 7.1 security hardening + Stage 7.2 occupancy enforcement + Stage 7.3 place_container_at_location + Stage 8B execution history convergence + Stage 8C canonical snapshot cleanup + Stage 8D legacy surface removal + Stage 9 PR1 placement caller repoint + Stage 9 PR2 facade route removal + Stage 9 PR3 SQL/RPC wrapper fate decision)

## How to read this

- `Owner` = the layer that must keep the invariant true
- `Enforced In` = real mechanisms only, not intent
- `Test Coverage` = current coverage in the repo, not aspirational coverage
- `Notes` = severity plus honesty about status: `implemented`, `partial`, `declared`, or `planned`

## Practical rule

When a new architecture invariant is introduced, it should be added here only after answering all four questions:

1. Who owns it?
2. Where is it enforced?
3. Which tests prove it?
4. Is it truly implemented, or only declared?
